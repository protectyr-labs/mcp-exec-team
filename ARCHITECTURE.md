# Architecture

Design rationale for the multi-persona debate engine.

## Why Sequential Turns Over Parallel

In a parallel system, every persona receives the same prompt and responds independently. The result is N disconnected monologues. Nobody builds on what someone else said. Nobody disagrees with a specific point.

Sequential turns fix this. Each persona sees the full transcript so far. The second speaker can reference the first. The third can synthesize or challenge both. This produces a conversation, not a survey.

The tradeoff is latency: N sequential API calls instead of N parallel ones. For a 3-persona roundtable, this means ~3x the wall-clock time. This is acceptable because the output quality is dramatically higher, and roundtables are not latency-sensitive operations.

## Why Sparse Reactions

In early experiments, asking every persona to react to every other persona produced a wall of "I agree with the engineer" noise. Real meetings don't work that way. Most people nod silently and only speak up when something is genuinely important.

The reaction system enforces this:
- Maximum 3 reactions per roundtable
- A persona never reacts to their own statement
- Reactions are generated by a separate observer pass, not by the personas themselves
- Only three reaction types: `agree`, `good-point`, `important`

This produces the occasional "Staff Engineer reacts to Security Lead: important" that adds signal without adding noise.

## Why the [PASS] Rule

Response bloat is the number one problem in multi-agent systems. When you ask 5 agents a question, you get 5 answers -- even when 3 of them have nothing useful to say. They'll restate the obvious, agree with what's already been said, or pad their response with caveats.

The [PASS] rule makes silence a first-class option. Every persona prompt includes the instruction: "If you have nothing important to add, respond with exactly [PASS] and nothing else. Silence is better than filler."

In practice, [PASS] triggers about 20-30% of the time in a 3+ persona roundtable. This is a feature, not a bug. It means the remaining responses are higher signal.

## Why War Room Constraints

The War Room mode constrains responses to 2-4 sentences with a single clear point. Without this, personas tend to write 2-3 paragraphs covering every angle they can think of. The constraint forces prioritization: what is the ONE thing this persona needs to say right now?

The four allowed response types are:
1. Propose a concrete option
2. Ask a specific question
3. Give one number or fact
4. Suggest a better alternative

This maps to how effective executives actually communicate in high-stakes meetings.

## Why 3 Personas in the Demo

Two personas is a conversation. Three or more is a meeting. The demo includes three (Product Manager, Staff Engineer, Security Lead) because:

- It demonstrates the sequential turn mechanic (persona 3 sees what 1 and 2 said)
- It shows natural tension (ship fast vs. build right vs. stay secure)
- It's small enough to understand quickly but large enough to be non-trivial
- The [PASS] rule is unlikely to trigger with only 2 personas but starts appearing with 3+

Users can add as many custom personas as they want. The engine handles N personas.

## Why Markdown Persona Files

Personas are markdown files, not JSON configs or database entries. This is deliberate:

- Markdown is human-readable and human-editable
- Persona files can be version-controlled with git
- No schema to learn -- just write a system prompt
- Files in a directory are self-discovering (no registry to maintain)

The persona file IS the system prompt. The engine reads it and passes it directly to the LLM. Whatever you write in the file is what the persona becomes.

## Known Limitations

1. **Requires Anthropic API key** -- The engine uses Claude as the underlying LLM. No support for other providers yet. Without an API key, responses are stubbed with placeholder text.

2. **Sequential means slower** -- A 5-persona roundtable makes 5 sequential API calls plus 1 reaction call. Expect 15-30 seconds depending on response length. This is inherent to the sequential design.

3. **No streaming** -- Each turn completes fully before the next begins. Streaming mid-turn would break the sequential guarantee.

4. **Reaction quality varies** -- The reaction generator is a separate LLM call that sometimes produces no reactions or occasionally misattributes them. The validation layer catches invalid reactions but can't guarantee semantic accuracy.

5. **No memory across debates** -- Each roundtable is independent. Personas don't remember what they said in previous debates unless you pass it as context.

6. **Single model** -- All personas use the same underlying model. There's no way to assign different models to different personas yet.
